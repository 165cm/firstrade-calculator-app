# 🤖 AI共通ルール

このファイルは Claude Code と Gemini CLI の両方が参照する共通ルールです。

---

## 📌 プロジェクト概要

- **言語**: TypeScript / JavaScript
- **開発者レベル**: 初心者（サポートが必要）
- **Git運用**: コミット＆プッシュのみ（プルリクエストは使わない）

---

## 🎯 最重要ルール（必ず守ること）

### 1. コードを提供するときは「完成版」で

```
❌ ダメな例: 「この部分を修正してください」
✅ 良い例:   ファイル全体のコードを提供する
```

### 2. 説明は超具体的に

```
❌ ダメな例: 「ターミナルでコマンドを実行」
✅ 良い例:   「ターミナルを開き、以下をコピペしてEnterキーを押す」
```

### 3. 専門用語は使わない or 説明をつける

```
❌ ダメな例: 「非同期処理でPromiseを返します」
✅ 良い例:   「非同期処理（時間のかかる処理を待つ仕組み）で〜」
```

## 📝 コーディングルール

### ファイル・フォルダ名

| 種類 | ルール | 例 |
|-----|--------|-----|
| ファイル名 | キャメルケース（小文字始まり） | `userProfile.ts` |
| コンポーネント | パスカルケース（大文字始まり） | `UserCard.tsx` |
| フォルダ名 | ケバブケース（ハイフン区切り） | `user-settings/` |

### コメントのルール

```typescript
// ✅ 関数には必ず「何をするか」を日本語で書く
/**
 * ユーザー情報を取得する
 * @param userId - ユーザーのID
 * @returns ユーザー情報のオブジェクト
 */
function getUser(userId: string) {
  // 処理の中身...
}
```

### 変数・関数名

```typescript
// ✅ 良い例: 何をするか分かる名前
const userName = "田中";
function calculateTotalPrice() { }

// ❌ ダメな例: 意味不明な省略
const u = "田中";
function calc() { }
```

## 💬 コミットメッセージのルール

### 基本フォーマット

```
種類: 何をしたか（日本語でOK）
```

### 使える種類（タイプ）

| タイプ | 意味 | 使う場面 |
|-------|------|---------|
| `feat` | 新機能 | 新しい機能を追加したとき |
| `fix` | バグ修正 | エラーや不具合を直したとき |
| `docs` | ドキュメント | READMEなどの説明文を変更したとき |
| `style` | 見た目 | コードの動きに関係ない変更（空白、インデントなど） |
| `refactor` | リファクタリング | 動作は変えずにコードを整理したとき |
| `chore` | 雑務 | 設定ファイルの変更など |

### 具体例

```bash
feat: ログインボタンを追加
fix: ユーザー名が表示されないバグを修正
docs: READMEにインストール手順を追加
style: コードのインデントを統一
refactor: ユーザー取得処理を関数に切り出し
chore: パッケージを更新
```

## 🚀 コミット＆プッシュの手順

### 毎回やること（この順番で！）

```bash
# ① 変更したファイルを準備する
git add .

# ② コミットする（変更を記録）
git commit -m "feat: ○○を追加"

# ③ プッシュする（GitHubにアップロード）
git push
```

### コミットのタイミング

- ✅ 1つの機能が完成したとき
- ✅ バグを1つ直したとき
- ✅ 作業を中断する前（バックアップとして）

### やってはいけないこと

- ❌ 何日も溜め込んでから一気にコミット
- ❌ 「update」「修正」など意味不明なメッセージ
- ❌ 動かないコードをプッシュ

## 🚨 エラーが起きたときの対応

AIは以下の形式でエラーを説明すること：

```
## ❌ エラー内容
（エラーメッセージを表示）

## 🔍 原因
（なぜこのエラーが起きたか、初心者向けに説明）

## ✅ 解決方法
（具体的な手順を番号付きで）

1. 〇〇を開く
2. △△をコピペする
3. Enterキーを押す
```

## 📁 ファイル管理ルール

### バージョン管理

- 古いコードは消さずにコメントアウトで残してもOK
- 大きな変更の前は「どのファイルを変更するか」を先に説明する

### どれが最新？と聞かれたら

- ファイル名と更新日時を明確に伝える
- 「このファイルが最新版です」と明言する

---

## 📞 コミュニケーションルール

1. **確認を取る**: 大きな変更の前は「〇〇してもいいですか？」と聞く
2. **選択肢を提示**: 複数の方法があるときは選択肢として提示
3. **進捗を報告**: 長い作業は途中経過を報告する
